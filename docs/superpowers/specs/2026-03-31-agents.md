# Agents (Track C)

**Goal:** Add named, project-scoped agent entities with their own provider, instructions, and run history. Agents can be assigned to tasks to run sessions with their specific configuration. A "Run with agent" flow on task detail enables per-task agent selection.

**Dependencies:** Track B (Providers) must be built first — agents reference `provider_id`.

---

## Data Model

### New `agents` table

```sql
CREATE TABLE agents (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  name TEXT NOT NULL,
  title TEXT,
  provider_id TEXT,
  model TEXT,
  instructions_path TEXT,
  status TEXT NOT NULL DEFAULT 'idle',
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
)
```

| Column | Notes |
|---|---|
| `provider_id` | Nullable FK → `providers.id`. Falls back to project → global resolution (Track B) if null. |
| `model` | Nullable. Overrides the provider's default model when set. |
| `instructions_path` | Relative path within the project directory, e.g. `.agents/ceo/`. The app manages this directory. |
| `status` | `idle` / `running` / `paused` |

### `agent_id` column on `sessions`

```sql
ALTER TABLE sessions ADD COLUMN agent_id TEXT
```

Nullable. Set when a session is spawned via "Run with agent". Enables the Runs tab to show which agent ran which session.

## Instructions on Disk

Each agent gets a managed directory at `{project.path}/.agents/{agent-slug}/` containing a single `instructions.md` file edited via the app UI.

At session spawn time, `writeInstructions(agent, providerType)` writes a provider-specific instruction file into the working directory so each CLI picks it up natively:

| Provider type | File written | How it's used |
|---|---|---|
| `claude` | `CLAUDE.md` (or `--system-prompt` flag) | Claude Code reads `CLAUDE.md` automatically |
| `codex` | `AGENTS.md` | Codex reads `AGENTS.md` natively |
| `gemini` | `GEMINI.md` | Gemini CLI reads `GEMINI.md` natively |
| `ollama` | Injected as system message via API | No file needed |

The instruction files are written to a temp location or the project root before spawn and cleaned up after. They are **not committed to git** (added to `.gitignore` if not already ignored).

## Agent Screen

### Agents list — `/projects/[projectId]/agents`

Grid or list of all agents for the project. Each card shows:
- Name + title
- Provider badge (e.g. "Claude Code")
- Status dot (idle / running / paused)
- Last active date

**Create agent** button opens a modal: name, title, provider (dropdown), model (optional text). Creates the agent record and initialises the `.agents/{slug}/instructions.md` file with a starter template.

### Agent detail — `/projects/[projectId]/agents/[agentId]`

Four tabs:

**Dashboard**
- Current status badge
- Last run: task title, timestamp, duration
- Total runs count
- Quick "Run heartbeat" button (future — placeholder for now)

**Instructions**
- Markdown editor for `.agents/{slug}/instructions.md`
- Save button — writes file to disk, no DB update needed
- File is shown with syntax highlighting

**Configuration**
- Name (text input)
- Title (text input)
- Provider (dropdown — lists active providers from Track B, plus "Inherit from project")
- Model (text input, optional — placeholder shows provider default)
- Save button — PATCHes the agent record

**Runs**
- List of sessions where `agent_id = this agent`. Newest first.
- Columns: task title (linked), phase, status, started, duration
- Empty state: "No runs yet"

**Skills tab** — placeholder, implemented in Track E.

## "Run with Agent" on Task Detail

On the task detail page (Track A), a **"Run with agent"** button sits alongside or replaces the existing phase-advance buttons.

Clicking opens a small popover/dropdown listing the project's agents with their provider badge. Selecting an agent:
1. Creates a session linked to the task at its current phase
2. Sets `session.agent_id = agent.id`
3. Calls `writeInstructions(agent, providerType)` before spawning
4. Resolves the provider using the agent's `provider_id` (Track B resolution chain)
5. Spawns the PTY process with the resolved binary + flags

The task detail's Live Runs section (Track A) picks up the new session automatically.
