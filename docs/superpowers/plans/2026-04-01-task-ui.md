# Task UI Improvements Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade the task detail view with a two-column layout (properties panel right, live terminal output left), agent todo checklist, and replace NewTaskModal with a richer CreateTaskModal supporting priority, labels, assignee, and a "Start now" flow.

**Architecture:** Three new columns (`priority`, `labels`, `assignee_agent_id`) are added to the `tasks` table via `ALTER TABLE` migrations in `lib/db.ts` and surfaced through `lib/db/tasks.ts`, the PATCH/POST API routes, and `hooks/useTasks.ts`. `TaskDetailView` is rewritten from a single-column layout into a two-column flex container: the left column hosts the editable title/description, a new `LiveRunsSection` component (which owns the WebSocket connection and TodoWrite parsing, previously scattered in `TaskDetailView`), and the agent todo checklist; the right 260px panel is a new `PropertiesPanel` component. The live-streaming logic is lifted entirely out of `RightDrawer`'s Sessions tab (which becomes history-only) and into `LiveRunsSection`. `NewTaskModal` is deleted and replaced by `CreateTaskModal` with the full field set plus "Start now". `TaskCard` gains a priority chip and `SessionAgentCard` gains a todo-count pill.

**Tech Stack:** Next.js App Router, React, SWR (useTasks), WebSocket, Vitest + @testing-library/react, inline styles

export async function POST(req: NextRequest) {

  })

export async function createTask(
  projectId: string,
  title: string,
  notes?: string,
  options?: { priority?: string; labels?: string[]; assignee_agent_id?: string | null }
): Promise<Task> {
  const res = await fetch('/api/tasks', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ projectId, title, notes, ...options }),
  })
  if (!res.ok) throw new Error('Failed to create task')
  const task = await res.json()
  await globalMutate(`/api/tasks?projectId=${projectId}`)
  return task
}

## Data Flow

**Task creation via "Start now":**
`CreateTaskModal` → `createTask(projectId, title, notes, { priority, labels })` → `POST /api/tasks` → `lib/db/tasks.createTask()` → SQLite INSERT with all fields → task returned → `POST /api/sessions` with `{ projectId, phase: 'brainstorm', taskId: task.id }` → `onNavigate(task.id)` → parent fetches task and sets `selectedTask`.

**Live runs + todo tracking:**
`TaskDetailView` mounts → renders `<LiveRunsSection taskId={task.id} onTodos={setTodos}>` → `LiveRunsSection` fetches `GET /api/sessions?taskId={taskId}&status=active` → active session found → `new WebSocket(.../api/sessions/{sessionId}/ws)` opened → each `ws.onmessage`: append to `logLines` (terminal output, capped at 500) + test TodoWrite regex → on match parse JSON array and call `onTodos(todos)` → `TaskDetailView.todos` state updated → agent-tasks checklist re-renders with per-status styling.

**Properties panel field edit:**
User changes a field in `PropertiesPanel` → `patchTask(task.id, { [field]: value })` → `PATCH /api/tasks/{id}` → `allowed` array check passes → `lib/db/tasks.updateTask()` → SQLite UPDATE → SWR `mutate` called for `/api/tasks/${taskId}` and all `/api/tasks?projectId=` keys → parent `useTask` hook re-fetches → `task` prop propagates to `TaskDetailView` → `PropertiesPanel` re-renders.

**Dashboard todo pill:**
`useOrchestratorFeed` produces `FeedEntry[]` per session → `SessionAgentCard` receives `feedEntries` prop → `parseTodos(feedEntries)` iterates from latest entry, matches `TodoWrite · [...]` regex → extracts todo array → `completedCount` computed → `{completedCount} / {todos.length}` pill renders next to Live badge; pill hidden when `todos.length === 0`.

## Critical Details

**Error handling:**
- All `patchTask` calls in `PropertiesPanel` and `TaskDetailView` must be wrapped in `.catch(console.error)` to prevent unhandled promise rejections from crashing the UI.
- `LiveRunsSection` WebSocket errors: `ws.onerror = () => setActiveSession(null)` to gracefully fall back to the inactive state.
- `CreateTaskModal` "Start now" flow: if `POST /api/sessions` returns 409 `concurrent_session`, log the existing session ID and still call `onNavigate(task.id)` — do not block the user.
- Label JSON parsing: always wrap `JSON.parse(task.labels)` in try/catch and default to `[]` on any parse error, in both `PropertiesPanel` and `CreateTaskModal`.

**State management:**
- `TaskDetailView` only keeps `title`, `description`, and `todos` in local state. All other task data lives in the SWR cache accessed via the `task` prop passed down from the parent page.
- `LiveRunsSection` re-fetches active sessions whenever it receives a `{ type: 'status', state: 'ended' }` WebSocket message, resetting `activeSession` to `null` and clearing `logLines`.
- `PropertiesPanel` reads directly from the `task` prop for all display values and does not maintain a local copy of priority/labels/assignee — SWR revalidation is the source of truth for current values.
- `logLines` in `LiveRunsSection` is capped at 500 entries to prevent unbounded memory growth: `setLogLines(prev => [...prev.slice(-499), newLine])`.

**Testing:**
- All tests that render components using WebSocket must stub `global.WebSocket` before render using the `MockWebSocket` class pattern from `LiveRunsSection.test.tsx`.
- Mock `@/hooks/useTasks` module-wide in all component tests; do not rely on SWR setup.
- API route tests use `initDb(':memory:')` and mock `getDb` to return that instance via `vi.mock('@/lib/db', ...)`.
- Run each task's tests in isolation before running the full suite: `npx vitest run <path> --reporter verbose`.
- When updating existing test fixtures for the new `Task` fields, search for all `Task` literal objects in `components/__tests__/` and add `priority: 'medium', labels: null, assignee_agent_id: null`.

**Performance:**
- `PropertiesPanel` agents fetch runs once on mount via `useEffect` with `[]` deps — not SWR — since the agents list changes infrequently.
- `LiveRunsSection` fetches the active session on mount only. It does not poll; the WebSocket `status: ended` message drives re-fetch logic.

**Security:**
- Strip or cap label values in `CreateTaskModal` and `PropertiesPanel` before calling `patchTask`: enforce max 32 chars, allow only `[a-zA-Z0-9-_ ]`.
- Validate `assignee_agent_id` in the PATCH handler: must be `null` or a non-empty trimmed string. Reject empty strings.
